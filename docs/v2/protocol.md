# `protocol` — CP ↔ Harness Wire Protocol

> **Depends on:** [`architecture.md`](architecture.md)
> **Owns:** the JSONL frame schema between control plane and per-agent harness processes. This is the contract every other component is written against.

---

## Transport

- **Medium:** stdio of the harness child process. CP writes to harness's stdin, reads from harness's stdout.
- **Format:** JSONL — one JSON object per line, terminated with `\n`. Partial trailing line on EOF is dropped.
- **Stderr:** human logs only. Never JSONL. CP captures stderr to per-agent log files (rotating).
- **Encoding:** UTF-8. Frames must not contain unescaped newlines (JSON-encoded strings handle this).
- **Backpressure:** OS pipe buffers handle flow control. If CP is slow reading, harness's `write` blocks. Treat as normal; do not buffer or drop.

---

## Frame envelope

All frames have a `type` discriminator. Frames participating in request/response pairs include a `request_id` (UUID v4 generated by sender; receiver echoes on response). Frames carrying conversation context include `conversation_id`.

**Convention:** `type` uses dot-namespacing. Top-level domains: `init`, `shutdown`, `user_message`, `inter_agent.*`, `cron_trigger`, `abort`, `compact`, `memory.*`, `agent.*` (subagent), `tool.*`, `channel.*`, `agent_event`, `compaction.*`, `transcript.*`, `ready`, `fatal`.

**Schema documentation rule:** every frame type below is canonical. If implementation ever needs to add a field, this spec is updated in the same PR. The spec is the contract.

## Inbound frames (CP → Harness, on stdin)

### `init`

Sent first, exactly once, before any other inbound frame. Harness must respond with `ready` (outbound) before processing further frames.

```jsonc
{
  "type": "init",
  "agent_id": "agent-uuid",
  "agent_name": "echo",
  "data_dir": "/home/user/.argos/agents/<id>",
  "home_dir": "/home/user/.argos/agents/<id>/home",
  "memory_root": "/home/user/.argos/agents/<id>/memory",
  "conversations_root": "/home/user/.argos/agents/<id>/conversations",
  "subagents_dirs": [
    "/home/user/.argos/agents/<id>/subagents",
    "/home/user/.argos/subagents"
    // built-in dir is bundled into harness package; not listed here
  ],
  "model": "claude-sonnet-4-6",
  "provider": "anthropic",
  "context_window": 200000,
  "compaction_threshold": 0.85,    // 0.0-1.0; defaults to 0.85
  "api_keys": { "anthropic": "...", "openai": "..." },
  "channels": [
    { "channel": "matrix",
      "config": {
        "homeserver": "https://matrix.org",
        "user_id": "@bot:matrix.org",
        "access_token": "...",
        "e2ee_store_path": "/home/user/.argos/agents/<id>/channels/matrix/store"
      } }
  ],
  "cron_tasks": [
    { "id": "task-uuid",
      "name": "morning_summary",
      "cron": "0 8 * * *",
      "timezone": "America/New_York",
      "prompt": "Summarize what's on my calendar today.",
      "conversation_id": "cron-task-uuid",
      "enabled": true }
  ],
  "internal_token": "placeholder-change-me",  // shared secret CP↔harness; not currently used (stdio is the trust boundary), reserved
  "log_level": "info",
  "is_subagent": false,             // true when this is a subagent run
  "parent_agent_id": null           // populated when is_subagent: true
}
```

**Notes:**

- `subagents_dirs` is in resolution order: most-specific (per-agent) first; built-ins are appended by the harness internally.
- `is_subagent`/`parent_agent_id`: present so the harness can adjust behavior (e.g., suppress channel adapter init, suppress git auto-push, run with `bypassPermissions`).

### `shutdown`

Graceful shutdown. Harness should flush pending writes, push MemFS, exit cleanly within `grace_ms`.

```jsonc
{ "type": "shutdown", "grace_ms": 5000 }
```

If harness doesn't exit within `grace_ms`, CP follows up with SIGTERM, then SIGKILL.

### `user_message`

A non-channel-sourced inbound message. Channels ingest messages internally in the harness; this frame is only for CLI / external gateway / future PWA / inter-agent.

```jsonc
{
  "type": "user_message",
  "conversation_id": "default",
  "content": "What's on my calendar today?",
  "source": "cli",                  // "cli" | "external_gateway" | "pwa"
  "idempotency_key": "uuid"         // optional; CP-generated for external_gateway
}
```

### `inter_agent.receive`

CP routes a message from another agent's harness here.

```jsonc
{
  "type": "inter_agent.receive",
  "conversation_id": "inter-<peer_agent_id>",
  "from_agent": "agent-uuid",
  "from_agent_name": "research-bot",
  "content": "I found three relevant papers, see /tmp/papers.json"
}
```

### `cron_trigger`

CP's scheduler fires a task; sends this frame.

```jsonc
{
  "type": "cron_trigger",
  "task_id": "task-uuid",
  "task_name": "morning_summary",
  "conversation_id": "cron-task-uuid",
  "prompt": "Summarize what's on my calendar today."
}
```

### `abort`

Stop currently-running agent loop for a specific conversation, or all if `conversation_id` omitted.

```jsonc
{ "type": "abort", "conversation_id": "default" }
```

### `compact`

Force compaction of a conversation. Used by `argos compact <agent>` CLI ops command and by sleeptime tooling.

```jsonc
{ "type": "compact", "conversation_id": "default" }
```

### `memory.write`

CP forwards a memory-block edit (originating from PWA / CLI). Harness writes the file, then commits and pushes (if part of MemFS auto-push policy). Request/response — harness must reply with `tool.response`.

```jsonc
{
  "type": "memory.write",
  "request_id": "uuid",
  "path": "system/persona",         // relative to memory_root, no .md extension
  "content": "..."
}
```

### `agent.event` (forwarded child events)

When a subagent's harness emits an `agent_event`, CP forwards it to the parent harness with this frame.

```jsonc
{
  "type": "agent.event",
  "agent_id": "subagent-uuid",      // the child's id
  "event": { /* AgentEvent payload, see below */ }
}
```

### `agent.result` (subagent finished)

Final result of a spawned subagent, delivered to the parent.

```jsonc
{
  "type": "agent.result",
  "request_id": "uuid",             // matches the parent's agent.spawn request_id
  "agent_id": "subagent-uuid",
  "ok": true,
  "result": "Summary of findings: ...",
  "error": null,                    // present if ok: false
  "total_tokens": 4520,
  "duration_ms": 8421
}
```

### `tool.response`

Response to a CP-mediated tool request the harness emitted (cron.add, agents.list, memory.write, etc.).

```jsonc
{
  "type": "tool.response",
  "request_id": "uuid",
  "ok": true,
  "result": { /* tool-specific shape */ },
  "error": null
}
```

### `channel.config_updated`

PWA/CLI changed channel config; harness reloads the affected adapter without restart.

```jsonc
{
  "type": "channel.config_updated",
  "channel": "matrix",
  "config": { /* same shape as init.channels[*].config */ }
}
```

## Outbound frames (Harness → CP, on stdout)

### `ready`

Emitted when init has been processed and all configured channel adapters have connected. CP marks the agent `ready` in its registry (this finally fixes the `markReady()` never-called bug from v1).

```jsonc
{ "type": "ready", "capabilities": ["matrix", "memory", "cron-tools"] }
```

### `fatal`

Unrecoverable error. Harness will exit immediately after emitting.

```jsonc
{ "type": "fatal", "message": "Failed to clone MemFS git remote: <reason>" }
```

### `agent_event`

Wraps a pi-agent-core `AgentEvent`. Tagged with conversation. CP routes / logs / forwards to subscribers.

```jsonc
{
  "type": "agent_event",
  "conversation_id": "default",
  "event": {
    // pi-agent-core's AgentEvent shape; see packages/agent/src/types.ts:374-389
    "type": "message_update",       // also: agent_start, agent_end, turn_start, turn_end,
                                    //   message_start, message_end, tool_execution_start,
                                    //   tool_execution_update, tool_execution_end
    "delta": "...",
    // ... other event-specific fields
  }
}
```

**Visual rendering (CLI/PWA):** when a `tool_execution_*` event references a tool with a path argument under `$MEMORY_DIR`, renderers display "Reading memory…", "Updating memory…", etc. instead of the raw tool name. Implementation lives in renderers, not tools.

### `compaction.start` / `compaction.complete`

Observability for compaction. CP logs; future PWA can display.

```jsonc
{ "type": "compaction.start",    "conversation_id": "default", "tokens_before": 174000 }
{ "type": "compaction.complete", "conversation_id": "default",
  "tokens_before": 174000, "tokens_after": 32000,
  "evicted_count": 142, "summary_id": "msg_01H..." }
```

### `inter_agent.send`

Agent invoked the `SendAgentMessage` tool. CP routes to target's harness via `inter_agent.receive`.

```jsonc
{
  "type": "inter_agent.send",
  "request_id": "uuid",             // optional; if present, sender awaits ack
  "to_agent": "agent-uuid-or-name",
  "conversation_id": null,          // null → use inter-<sender_id> on receiver
  "content": "Have you seen the new staging deploy?"
}
```

### `agent.spawn`

Agent invoked the `Agent` tool. CP spawns a child harness, configured per the registry entry for `agent_type`.

```jsonc
{
  "type": "agent.spawn",
  "request_id": "uuid",             // parent's tool waits on this
  "agent_id": "subagent-uuid",      // generated by parent harness
  "agent_type": "recall",           // resolves via subagents_dirs
  "task": "Search transcripts for 'argos migration'. Return a summary.",
  "model_override": null,
  "tools_override": null,
  "inherit_memory": false           // false = isolated (read-only access to parent's memory by default)
}
```

### `tool.request`

CP-mediated tool — agent invoked a tool that needs CP state, harness asks CP to do it.

```jsonc
{
  "type": "tool.request",
  "request_id": "uuid",
  "tool": "cron.add",               // also: cron.remove, cron.list, agents.list, channel.send_dm, memory.write_proxy, etc.
  "args": { /* tool-specific shape */ }
}
```

CP processes, sends `tool.response` (inbound) with the result.

### `transcript.appended`

Notify CP that a new transcript entry landed. Cheap (~100-200 bytes), one per canonical message boundary (not per streaming chunk). Enables future PWA "live update" without polling.

```jsonc
{
  "type": "transcript.appended",
  "conversation_id": "default",
  "message_id": "msg_01H...",
  "kind": "user" | "assistant" | "tool_call" | "compaction",
  "ts": "2026-05-05T20:30:00Z"
}
```

## Request / response correlation

For frames that need a response (`memory.write`, `inter_agent.send` (when `request_id` set), `agent.spawn`, `tool.request`):

1. Sender generates UUID v4 as `request_id`.
2. Sender writes the frame.
3. Sender awaits the corresponding response frame with matching `request_id`.
4. Receiver processes, generates response frame, includes the same `request_id`.
5. Sender resolves the awaiting promise on receipt.

**Timeout:** sender's responsibility. Default suggested: 30s for memory.write, 60s for tool.request, 10min for agent.spawn (subagents may run a while), no timeout for inter_agent.send (fire-and-forget if `request_id` omitted).

**Out-of-order responses:** acceptable. Sender keys an in-memory map by `request_id`, resolves whichever arrives.

## Conversation ID namespacing

| Prefix | Source | Notes |
|---|---|---|
| `default` | PWA, CLI, primary thread | One per agent |
| `tg-<chat_id>` | Telegram (post-v1) | One per Telegram chat |
| `matrix-!<room_id>:<server>` | Matrix | One per Matrix room |
| `inter-<peer_agent_id>` | Inter-agent peer messaging | One per peer pair |
| `cron-<task_id>` | Cron-triggered turns | One per cron task |
| `ext-<request_id>` | External Gateway one-shot (post-v1) | Ephemeral; CP returns response synchronously |

The harness creates conversation directories on first reference. CP's stdin frames either reference an existing conversation or implicitly create it.

## Versioning

- This document version is the protocol version. v1 of the protocol = the schema above.
- Breaking changes bump the version. v2-and-up include `protocol_version` in `init`; harness must reject unknown versions.
- For v1 of v2: no `protocol_version` field needed — both CP and harness ship from the same monorepo and version-lock.

---

## Compatibility with pi-agent-core's AgentEvent

The `event` payload inside our `agent_event` frame is **passed through verbatim** from pi-agent-core. We don't re-wrap or normalize. Renderer implementations should consume pi-agent-core's `AgentEvent` shape (`packages/agent/src/types.ts:374-389`) directly.

If pi-agent-core changes its event shape in a future version, our pi-agent-core dependency is pinned and that change is opt-in via dependency upgrade (which would be a single PR with renderer updates).

## Open considerations

- **Frame size limit.** Pipes have an OS-dependent atomic-write size (`PIPE_BUF`, often 4096 or 65536). Frames larger than this can interleave with other writes. We should ensure single-writer per direction (one harness has one stdout writer), so this isn't a real concern. If we ever multiplex (we don't), we need framing.
- **Backpressure on burst events.** A streaming text response can produce many `agent_event` frames per second. If CP's reader stalls (e.g., disk write to a log file blocks), harness's `write` blocks. This is correct flow control but could pause the agent loop. Mitigation: CP should consume eagerly and queue downstream work asynchronously. To be revisited if observed.
- **Schema evolution within v1.** Additive changes (new optional fields) are non-breaking. Removing/renaming fields requires bumping protocol version. Be conservative.
